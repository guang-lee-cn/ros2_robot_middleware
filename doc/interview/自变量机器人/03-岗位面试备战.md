## 三、高频面试题（按 JD 匹配 + 我们的答案）

### 第 1 层：C++ 工程基座

#### Q1: shared_ptr vs unique_ptr 在 ROS2 中的使用场景？

**答案：**
> ROS2 中所有通信对象（Publisher/Subscription/Service/Action Client）都是 shared_ptr。因为 DDS 层内部有多个引用（Node 持有、Executor 持有、DDS participant 持有）。我用 shared_ptr 管理跨模块的生命周期——比如 LidarNode 的 publisher 被 LifecycleNode 和 DDS writer 同时引用，只有两者都释放才真正销毁。
>
> unique_ptr 用于独占资源——我的 KalmanFilter2D 类内部用 C 数组 double[4] 存状态向量，不存在共享需求。

#### Q2: 虚函数与多态 — 你的 LifecycleNode 模式用到了吗？

**答案：**
> 我们的 7 个节点全部继承 `rclcpp_lifecycle::LifecycleNode`，重写 5 个虚函数回调（on_configure/on_activate/on_deactivate/on_cleanup/on_shutdown）。这是模板方法模式——基类定义了状态机的框架（configure→activate→deactivate→cleanup），派生类实现具体行为。和 5G 基站的板卡状态机完全同构。

#### Q3: 你的项目中遇到过死锁吗？

**答案：**
> 目前没有，因为我们的 executor 模型是单线程（SingleThreadedExecutor）——所有回调顺序执行，不存在多线程竞争。但我知道风险在哪里：M3.3 提到如果为了性能切换到 MultiThreadedExecutor，sensor callback 和 timer callback 同时访问 KalmanFilter2D 的 state_ 和 cov_ 就会产生数据竞争。解决方案是加 mutex 或者把 KF 的 predict/update 放到同一个 callback group。

#### Q4: C++17 在你的项目中怎么用的？

**答案：**
> `constexpr` 代替宏定义（static constexpr int kNumNodes=6）、结构化绑定（for const auto& [id, label] : sequence）、range-for（for double& v : state_ v=0.0）、`if constexpr` 在 KF 中做编译期分支、`std::array` 代替 C 数组做编译期大小检查。

### 第 2 层：通信 & 集成

#### Q5: ROS2 Topic vs Service vs Action，什么时候用哪个？

**答案（面试 30 秒版）：**
> Topic 是发布-订阅，一对多异步数据流——传感器数据用 Topic（LiDAR 10Hz, IMU 100Hz）。Service 是一对一同步请求-响应——参数设置用 Service（SetParam）。Action 是带反馈和取消的异步任务——电机控制用 Action（MoveToPose 的 Goal→Feedback→Result）。
>
> 面试追问"为什么电机不用 Topic"：Topic 不支持取消——如果新感知数据到达需要立刻中止旧目标，Topic 做不到。Action 的 `async_cancel_goal` + preemption 模式是机器人导航的标准做法。

#### Q6: DDS QoS 你是怎么调优的？

**答案（30 秒版）：**
> 我按 topic 做了差异化 QoS。Camera 900KB 大 payload 用 best_effort + 2MB socket buffer——丢一帧不影响感知。IMU 100Hz 增量数据用 reliable——丢一帧积分链会断。心跳用 reliable——丢失必须被感知。我还在 `fastdds_profiles.xml` 里做了 per-participant 的 initialAnnouncements=5 加速多节点 discovery。

#### Q7: 你的项目里怎么处理低延迟通信？

**答案：**
> 两个层面。DDS 层：reliable 的 RTPS ACK/NACK 协议在 UDP 上实现了应用层重传，比 TCP 的重传更快（没有拥塞控制，适合实时场景）。应用层：KF 用实际 dt 而非硬编码 200ms——如果 timer 因为系统负载延迟到了 300ms，预测步的 dt=0.3 可以正确补偿，避免位置估计漂移。

#### Q8: 你怎么设计机器人内外部通信架构？

**答案（结合 JD-3）：**
> 内部通信：所有 7 个节点在同一个 DDS domain 内通过 Topic/Service/Action 通信。这本身就是 ROS2 的去中心化消息中间件——不需要额外的 MQTT broker 或 Kafka。外部通信：HealthMonitor 的 Prometheus HTTP endpoint 暴露给运维系统（Grafana），Fleet Manager 通过 DDS Partitions 跨 AMR 域通信——可以和云端协同通信对接。

### 第 3 层：平台思维 + 项目难题

#### Q9: 你项目中最大的技术难题是什么？怎么解决的？

**答案（核心——面试必问）：**
> 三个难题：
>
> **1. C++ name hiding 导致 LifecycleNode 编译失败**：派生类的 `create_publisher()` 私有方法隐藏了基类的 `create_publisher<T>()` 模板。修复用了两个策略：方法重命名（`create_report_publisher`）和全局域限定（`::shutdown` 避免隐藏 LifecycleNode::shutdown）。这暴露了 ROS2 API 的设计缺陷——`create_*` 命名太通用，派生类很容易意外覆盖。我写了 ADR 记录了这个坑。
>
> **2. 降级测试中 spin 和 publish 的竞态**：`rclcpp::sleep_for` 不驱动 executor，导致测试中 callback 从不执行，降级检测逻辑完全不触发。根因是 LifecycleNode 的 timer 和 subscription 都需要 executor spin 才能处理。修复是改用 spin_until + 手动 spin loop，并且 publish 必须在 spin_once 之前——否则 timer 在 subscription callback 之前评估，传感器全部被误判为 stale。
>
> **3. 多 AMR 命名空间隔离**：ROS2 的 namespace 机制对绝对 topic 路径不生效——`/sensor/lidar/heartbeat` 在 namespace `/amr1` 中仍然是 `/sensor/lidar/heartbeat` 而非 `/amr1/sensor/lidar/heartbeat`。生产环境的解决方案是用 ROS_DOMAIN_ID 做 DDS 级隔离（每台 AMR 独立 domain），而不是靠 ROS2 namespace。这是 DDS 的网络分区能力，比 namespace 方案更根本。

#### Q10: 你怎么把解决方案抽象为通用能力（SDK 思维）？

**答案：**
> 三个例子：
> 1. **KalmanFilter2D** 独立为头文件库，任何节点 include 就能用——无 ROS2 依赖，纯 C++17 数学库。这体现了"通用化"的思路。
> 2. **LifecycleNode 模式**：7 个节点用同一个 5 回调模板。如果新加一个 GPS 节点，只需实现 `on_configure` 创建 sub/pub，`on_activate` 启动 timer——框架已经固化。
> 3. **sensor_msgs 标准化**：传感器接口用 ROS2 标准消息类型，任何开源驱动（sick_scan2/realsense-ros）可以直接对接——不需要写适配层。
