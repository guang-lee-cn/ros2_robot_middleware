# 自变量机器人 技术一面 — 传感器降级

## 高频问题

### Q1: 你的 `fuse_no_lidar` 函数里，Camera 投影到 2D 用的公式是什么原理？精度为什么从 0.01m 降到 0.5m？

**考点：** 透视投影 vs 测距原理的理解深度

**答案要点：**
- Camera 是透视投影，没有深度信息——单张 RGB 图像无法直接获取距离
- 我的简化实现假设地面平坦 + 暗色像素 = 物体（强度分割），实际精度取决于场景纹理
- 0.5m 是保守估计——单目视觉测距在 5m 距离上的理论误差约 10%
- 真实场景应该用 depth-from-mono（深度学习）或 stereo matching，精度可到 0.05m
- 这是"有比没有好"的工程权衡——系统不因 LiDAR 挂了就停机

**反问面试官的点：**
- 贵司的 AMR 在传感器降级时是继续工作还是立即停机？
- 降级模式下的安全边界怎么定义？（比如降级后最大速度是否自动限制？）

### Q2: `evaluate_degradation()` 里 `missing >= 2` 直接返回 CRITICAL。如果 IMU + Camera 都挂了但 LiDAR 还在，为什么也算 CRITICAL？

**考点：** 传感器之间的依赖关系和融合逻辑的理解

**答案要点：**
- LiDAR 是唯一的主感知源——它可以独立完成距离测量
- 但 IMU 缺失 → 高频运动补偿丢失（100Hz → 10Hz），LiDAR 帧间匹配精度下降
- Camera 缺失 → 视觉分类不可用，无法区分人/柱子/叉车
- 两个辅助源同时挂掉 → LiDAR 在工作但运动补偿不准 + 无语义信息
- CRITICAL 通知 Decision 层：建议只做紧急避障，不建议继续导航任务
- 和 5G 基站的降额逻辑一致——单板故障继续服务，多板故障降额运行

**反问面试官的点：**
- 贵司的导航系统在 CRITICAL 降级下是立即停车还是降速？
- 行为树的 recovery 节点怎么处理感知层降级？

### Q3: 心跳字符串从 `"alive"` 变成了 `"degraded_no_lidar"` — HealthMonitor 现在能区分这些状态吗？

**考点：** 心跳协议的扩展性和监控层的消费逻辑

**答案要点：**
- 当前 HealthMonitor 只检测心跳是否超时（`last_seen` vs `timeout`），不解析字符串内容
- 需要改 `check_health()` 加入状态字符串解析
- `"degraded_*"` → status = WARN
- `"critical"` → status = ERROR
- `"alive"` → status = OK
- 这是控制面和数据面的分离——心跳的"有没有"和"是什么状态"是两个独立维度
- std_msgs/String 的扩展性优势在这里体现：加状态标签只需要改一行赋值，不需要改消息定义

**反问面试官的点：**
- 贵司的心跳协议包含哪些维度？（存活 / 负载 / 降级级别 / 版本号）
- 有没有遇到过心跳正常但节点业务逻辑卡死的场景？怎么检测？

## 代码走读高频追问

### Q4: `is_stale()` 判断数据过期的 timeout 值怎么选的？

**考点：** 传感器频率和超时阈值的关系

**答案：**
| 传感器 | 发布频率 | timeout | 容忍的丢帧数 |
|--------|---------|---------|------------|
| LiDAR | 10Hz (100ms) | 1.5s | 15 帧 |
| IMU | 100Hz (10ms) | 0.5s | 50 帧 |
| Camera | 5Hz (200ms) | 3.0s | 15 帧 |

- 超时值 = 发布周期的 N 倍，N 的选择取决于 QoS 和丢帧容忍度
- best_effort 的 LiDAR/Camera 丢帧率更高，所以 timeout 更大
- reliable 的 IMU 理论上不丢帧（重传保证），timeout 可以设短
- 1.5s 是工程经验值——网络抖动通常 1s 内恢复，超过就是真正的硬件故障

### Q5: `fuse_no_camera` 里直接调 `fuse_full`，和 FULL 的区别只在心跳字符串？这不就是什么都没做吗？

**考点：** 诚实承认当前实现的局限 + 展示扩展规划

**答案：**
- 当前代码里 `fuse_full` 确实不看 camera 数据——camera 只是作为一个 flag
- NO_CAMERA 和 FULL 的聚类逻辑一样是**有意为之**，因为当前 camera 数据只用于"M3 的视觉分类"，还没实现
- 区别表现在两个地方：心跳字符串（供监控系统）、对象 ID 前缀（供调试）
- M3 加入 YOLO/目标检测后，NO_CAMERA 路径会去掉分类信息，对象从 `"person_0"` 变成 `"unknown_0"`
- 面试时要诚实："当前版本的 camera 只是一个占位，M3 会加入实际的视觉分类"
