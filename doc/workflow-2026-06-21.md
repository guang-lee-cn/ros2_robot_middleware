## 第 2 课：DDS 基础 + QoS 机制

**核心收获：**

- DDS reliable 靠 RTPS 层 ACK/NACK 重传，不是磁盘持久化
- KEEP_LAST 驱逐的是最旧数据，不是最新
- DDS 不做流控——不因 consumer 慢而降低 producer 速率
- 重传失败条件：writer cache 不再持有请求的 seq → sample_lost
- IMU 100Hz + depth=10 的隐患：回调整体慢会导致最早未消费数据被驱逐
- Camera（绝对测量）vs IMU（增量测量）→ QoS 选择依据
- 累积误差是选 reliable 的根本原因

**待验证（M3）：**

- best_effort 下 camera 900KB payload 的实际丢帧率
- reliable 下 IMU 100Hz 的端到端延迟
- depth=10 够不够（定量计算）

## 第 3 课：Lifecycle 状态机精讲

**核心收获：**

### 状态机

- 5 个 Primary State：Unconfigured / Inactive / Active / ErrorProcessing / Finalized
- 7 个 Transition：configure(T10) / cleanup(T11) / activate(T12) / deactivate(T13) / shutdown(T5,T6,T14)
- Finalized 是吸收态（sink state），不可逆
- shutdown 是万能出口，从任意活着的主状态直达 Finalized
- 因此 `on_shutdown` 必须自包含所有清理逻辑（幂等性要求）

### 各回调职责分离

| 回调 | 做什么 | 不做什么 |
|------|--------|---------|
| `on_configure` | 创建通信对象（pub/sub/srv/action） | 不创建 timer，不启动工作 |
| `on_activate` | 创建 timer，激活 publisher | 不创建新的通信对象 |
| `on_deactivate` | 停 timer，关外部资源（socket/thread），停 publisher | 不销毁通信对象 |
| `on_cleanup` | 销毁通信对象，清空状态数据 | 不操作 timer（deactivate 已停） |
| `on_shutdown` | 自包含执行 deactivate+cleanup 的全部清理 | 假设前一步可能没执行过 |

### C++/POSIX 陷阱

- C++ name hiding：派生类任意同名方法会隐藏基类所有同名方法（含模板）
  - `create_publisher()` 隐藏了 `rclcpp::Node::create_publisher<T>()`
  - `shutdown()` 隐藏了 POSIX `::shutdown(int,int)`
  - 修复：方法重命名（`create_report_publisher`）、全局域限定（`::shutdown`）
- `LifecycleNode` 不是 `rclcpp::Node` 的直接子类——`using` 声明不可用
- `shutdown(fd)` 和 `close(fd)` 的区别：shutdown 强制唤醒阻塞线程，close 只释放 fd
- `prom_socket_ = -1` 哨兵值保证幂等性——同一个操作执行多次安全

### 测试影响

- LifecycleNode 的测试必须先 `configure()` + `activate()`
- 外部 `create_publisher` 返回 LifecyclePublisher，需要 `on_activate()` 才能发布
- 9 个 GWT 测试全部适配成功

## 第 4 课：7 节点逐个走读 + 代码逐行精读（进行中）

### LidarNode（已完成）

**timer_callback() —— 模拟 SICK TiM781：**

```cpp
msg.ranges[i] = 2.0F + 1.5F * std::sin(angle * 3.0F) * std::cos(angle * 2.0F);
msg.intensities[i] = 1.0F - msg.ranges[i] / 10.0F;
```

- 基准距离 2.0m + sin·cos 乘积产生 [0.5, 3.5] 的波动图案
- 相邻采样点之间有几何连续性——用三角函数而不是 rand() 来模拟真实 LiDAR
- Intensity 和 range 负相关（越远反射越弱）

**两个独立 timer 的设计理由：**

1. **故障隔离**：数据回调卡住不会延迟心跳，HealthMonitor 不会误报
2. **职责分离**：改心跳频率不用碰数据发布代码
3. **降级安全（M2）**：可以停数据 timer 但保留心跳 timer

面试话术："心跳是 liveness check，必须和数据发布路径解耦。这和分布式系统的控制面/数据面分离是一个道理。"

### FusionNode（已完成）

**传感器回调（lidar/imu/camera）：** 只做一件事——把最新数据存进缓存（SharedPtr copy）

**timer_callback() 融合算法 —— 一维扫描线聚类：**

```
LiDAR 360个点，角度从 -π 到 π，逐点扫描：

1. 判断 hit: range ∈ (0.1, 3.0)
   - range < 0.1 → 传感器盲区噪点，过滤
   - range > 3.0 → 超出检测能力，过滤

2. 状态机扫描：
   hit && start < 0  → 记录聚类起点
   !hit && start >= 0 → 聚类段结束
     width > 5  → 有价值的物体（提取中点，极坐标→笛卡尔）
     width ≤ 5  → 噪声，丢弃

3. 极坐标 → 笛卡尔：
   x = range * cos(angle), y = range * sin(angle)
```

**关键常数：**

| 参数 | 值 | 理由 |
|------|-----|------|
| kMaxObjects | 5 | DecisionNode 只取 objects[0] |
| kRangeThreshold | 3.0m | 仓库 AMR 检测范围 |
| kClusterGap | 5 点 | 过滤 <5 个连续点的噪声 |
| timer 周期 | 200ms | 5Hz 融合，匹配最慢的 Camera 5Hz |

**当前局限（M2 待改）：**
- 前置守卫要求 lidar + imu + camera 三者都存在才能融合
- IMU/Camera 数据仅当 flag，未参与实际位置计算
- 无时间戳对齐——Camera 旧帧和新 LiDAR 帧直接融合

### DecisionNode（已完成）

**`on_perception()` 关键发现：**

`wait_for_action_server(1s)` 是同步阻塞调用——阻塞 executor 线程 1 秒无法处理任何其他回调（新 perception、心跳）。Demo 阶段可接受（MotorCtrl 总是先启动），M2 应改为纯异步模式。

面试话术："ROS2 Action 的最佳实践是不 wait、直接发——`async_send_goal` 本身就是异步的。同步 wait 在当前场景是空开销，但扩展性不好。"

### MotorCtrlNode（进行中）

**`execute()` 函数读点：**
- 10Hz 控制循环（`rclcpp::Rate rate(10)`）
- 每次迭代检查 cancel → 检查到达 → 插值前进
- `step_size_` = 0.05m，由 SetParam service 可动态调整
- 反馈参数：当前位置、剩余距离、完成百分比

### QoS RxO 协商（跨节点知识点）

Publisher 和 Subscriber QoS 不匹配时，DDS 采用 **RxO（Requested vs Offered）协商**：
- 结果取最弱约束——Publisher 的 reliability 是天花板
- Publisher best_effort + Subscriber reliable → 最终 best_effort
- 面试话术："DDS 以生产者为准，因为生产者承担序列化和发送成本。这和 HTTP Accept/Content-Type 协商逻辑相反。"

## M2.1+M2.4: 传感器降级 + 故障注入测试

### 降级状态机设计

| 级别 | 条件 | 行为 |
|------|------|------|
| FULL (0) | 3 个传感器都有新鲜数据 | 正常融合 |
| NO_LIDAR (1) | LiDAR 超时 1.5s | Camera 投影到 2D 做伪 LiDAR |
| NO_CAMERA (2) | Camera 超时 3.0s | 纯 LiDAR 聚类，无视觉分类 |
| NO_IMU (3) | IMU 超时 0.5s | LiDAR 聚类 + 0.05m 漂移模拟 |
| CRITICAL (4) | ≥2 个传感器缺失 | 发布空结果，Decision 停止前进 |

### 关键实现细节

**数据新鲜度检测：**
```cpp
bool is_stale(Time last_update, double timeout_s) const {
    return (now() - last_update).seconds() > timeout_s;
}
```
每个 sensor callback 记录接收时间戳，timer 每 200ms 评估一次。超时值 = 发布周期 × N，N 的选择取决于 QoS（best_effort 丢帧率高，timeout 需设大）。

**心跳状态同步：**
- `"alive"` / `"degraded_no_lidar"` / `"degraded_no_camera"` / `"degraded_no_imu"` / `"critical"`
- 利用 std_msgs/String 的扩展性——加状态标签只需改一行赋值，不改消息定义

**降级过渡回调：**
- 每次 degrade/upgrade 打 WARN 日志
- 从 CRITICAL 恢复到单传感器缺失是正常行为（网络抖动 → 恢复）

### 调试 bug 记录

**Bug 1：`rclcpp::sleep_for` 不驱动 executor**
- 现象：降级测试全部失败，degradation_level 始终为 FULL
- 根因：sleep_for 只阻塞当前线程，不处理 DDS 事件/回调/timer
- 修复：改用 `spin_until` 或手动 `exec.spin_once()` loop

**Bug 2：订阅回调积压**
- 现象：LiDAR 缺失测试跳变到 CRITICAL 而非 NO_LIDAR
- 根因：每 100ms spin_once 一次但 publish 也更频繁，callback 无法及时处理 IMU 数据
- 修复：spin 间隔缩短到 50ms，publish 间隔拉长到 250ms（< 500ms IMU timeout）

**Bug 3：publish 和 spin 的顺序**
- 现象：第一个 timer 评估时 sensor 数据还未被 callback 处理
- 根因：先 spin 再 publish → timer 在 callback 处理前评估 → 所有 sensor 都 stale
- 修复：先 publish 再 spin_once → callback 先处理新鲜数据 → timer 后评估
