## 第 2 课：DDS 基础 + QoS 机制

**核心收获：**

- DDS reliable 靠 RTPS 层 ACK/NACK 重传，不是磁盘持久化
- KEEP_LAST 驱逐的是最旧数据，不是最新
- DDS 不做流控——不因 consumer 慢而降低 producer 速率
- 重传失败条件：writer cache 不再持有请求的 seq → sample_lost
- IMU 100Hz + depth=10 的隐患：回调整体慢会导致最早未消费数据被驱逐
- Camera（绝对测量）vs IMU（增量测量）→ QoS 选择依据
- 累积误差是选 reliable 的根本原因

**待验证（M3）：**

- best_effort 下 camera 900KB payload 的实际丢帧率
- reliable 下 IMU 100Hz 的端到端延迟
- depth=10 够不够（定量计算）

## 第 3 课：Lifecycle 状态机精讲

**核心收获：**

### 状态机

- 5 个 Primary State：Unconfigured / Inactive / Active / ErrorProcessing / Finalized
- 7 个 Transition：configure(T10) / cleanup(T11) / activate(T12) / deactivate(T13) / shutdown(T5,T6,T14)
- Finalized 是吸收态（sink state），不可逆
- shutdown 是万能出口，从任意活着的主状态直达 Finalized
- 因此 `on_shutdown` 必须自包含所有清理逻辑（幂等性要求）

### 各回调职责分离

| 回调 | 做什么 | 不做什么 |
|------|--------|---------|
| `on_configure` | 创建通信对象（pub/sub/srv/action） | 不创建 timer，不启动工作 |
| `on_activate` | 创建 timer，激活 publisher | 不创建新的通信对象 |
| `on_deactivate` | 停 timer，关外部资源（socket/thread），停 publisher | 不销毁通信对象 |
| `on_cleanup` | 销毁通信对象，清空状态数据 | 不操作 timer（deactivate 已停） |
| `on_shutdown` | 自包含执行 deactivate+cleanup 的全部清理 | 假设前一步可能没执行过 |

### C++/POSIX 陷阱

- C++ name hiding：派生类任意同名方法会隐藏基类所有同名方法（含模板）
  - `create_publisher()` 隐藏了 `rclcpp::Node::create_publisher<T>()`
  - `shutdown()` 隐藏了 POSIX `::shutdown(int,int)`
  - 修复：方法重命名（`create_report_publisher`）、全局域限定（`::shutdown`）
- `LifecycleNode` 不是 `rclcpp::Node` 的直接子类——`using` 声明不可用
- `shutdown(fd)` 和 `close(fd)` 的区别：shutdown 强制唤醒阻塞线程，close 只释放 fd
- `prom_socket_ = -1` 哨兵值保证幂等性——同一个操作执行多次安全

### 测试影响

- LifecycleNode 的测试必须先 `configure()` + `activate()`
- 外部 `create_publisher` 返回 LifecyclePublisher，需要 `on_activate()` 才能发布
- 9 个 GWT 测试全部适配成功
