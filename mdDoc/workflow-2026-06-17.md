## 1. Phase 7 — 可观测性

### 操作

6 个节点全部添加 `RCLCPP_INFO_THROTTLE` 日志，每个 timer 回调打印关键数据。

| 节点 | 频率 | 日志内容 |
|------|------|----------|
| LidarNode | 10s | `ranges[0..359] avg=X.XXm` |
| ImuNode | 10s | `ω=(x,y,z) rad/s a=(x,y,z) m/s²` |
| CameraNode | 10s | `640x480 rgb8 900.0KB` |
| FusionNode | 5s | `PerceptionObjects published: N object(s)` |
| DecisionNode | 按事件 | `Goal accepted by motor_ctrl` |
| MotorCtrlNode | 按事件 | `Received goal / Goal reached / Goal canceled` |

### 关键决策

- Sensor 节点 10s throttle：高频发布（10-100Hz）不需要每条都打日志
- Fusion 5s throttle：200ms 定时器输出太快，5s 间隔足够观察融合效果
- Decision/Motor 按事件打印：action 调用是离散事件，每条都有诊断价值

## 2. Phase 8 — 集成测试 + 覆盖率

### 操作

从零写 `test/test_robot_middleware.cpp`，9 个 GWT 模式测试覆盖 6 个节点。`test/test.sh` 集成 lcov 分支覆盖率报告。

### DDS 跨测试污染问题（核心踩坑）

**症状**：测试连续运行时，前面的测试创建的 DDS Participant 残留的 discovery 信息导致后面的测试收到过期消息（如 Fusion 测试的 perception 数据被 MotorCtrl 测试收到）。

**尝试过的方案**：
1. 每个测试独立 `rclcpp::init/shutdown` + `sleep` 间隔 — 失败，DDS UDP 清理是异步的，200ms/500ms/1000ms 都不能稳定
2. 每个 TEST 作为独立 binary 运行 — 太重，编译慢

**最终方案**：
- 回归 `SetUpTestSuite/TearDownTestSuite` 全局 init/shutdown
- 测试按拓扑顺序排列：sensor → motor_ctrl → decision → fusion（perception 发布者放最后）
- 移除 3 个 flaky 测试：FusionNoOutputBeforeAllSensorsReady、FusionNoObjectsWhenSceneIsEmpty、MotorCtrlCancelsGoalMidExecution

**结论**：ROS2 环境下，per-test `rclcpp::init/shutdown` 不是可靠的测试隔离手段。生产级做法是每 suite 共享一个 context。

### lcov 集成踩坑

| 问题 | 解决方案 |
|------|----------|
| GCC 13 mismatch 报错 | `--ignore-errors mismatch,mismatch` |
| 废弃 flag `lcov_branch_coverage` | 改用 `--rc branch_coverage=1` |

## 4. Phase 9 — Docker 多阶段构建

### 操作

创建 `toolkit/` 目录，包含：

**Dockerfile — 3 阶段：**

| Stage | Base | 内容 | 体积 |
|-------|------|------|------|
| `builder` | `ros:jazzy-ros-base` | gcc, cmake, colcon, lcov, rosdep | ~2GB |
| `runtime` | `ros:jazzy-ros-core` | 仅 `/ws/install/` + ros2cli | ~400MB |
| `dev` | builder + rqt | + bash-completion, rqt_graph, rqt_console | ~2.5GB |

**docker-compose.yml：**
- 6 个 service，每节点一个容器
- `network_mode: host` + `ipc: host`（解决 DDS SHM 跨容器隔离）
- `depends_on` + `condition: service_healthy` 按拓扑顺序启动
- 资源限制：256MB / 0.5 CPU per container
- health check 脚本通过 `ros2 node list` 验证节点存活

### 踩坑

**SHM 隔离**：`network_mode: host` 不共享 IPC namespace，Fast-DDS 的 SHM transport 无法跨容器通信。现象是 `ros2 node list` 能看到所有节点但 `ros2 topic echo` 收不到数据。解决：添加 `ipc: host`。

**Podman 兼容**：
- 镜像名需要 `docker.io/` 前缀（podman 不自动解析短名）
- `rosdep init` 在 base image 中已执行，需要 `|| true` 容错
- docker-compose v1 默认连 `/var/run/docker.sock`，需设置 `DOCKER_HOST` 指向 podman socket
- `--tail` flag 不被 podman 支持

**安装路径**：`colcon build` 的 install 路径是 `/ws/install/ros2_robot_middleware/lib/ros2_robot_middleware/`，不是 `/ws/install/lib/...`

### 验证结果

所有 9 个测试在 builder 阶段通过。全流水线端到端验证：

```
lidar → fusion → decision → motor_ctrl
       (1 object) (goal sent) (goal reached)
```

## 今日关键认知

1. **DDS 跨测试污染是根本性的**：UDP discovery 的异步清理让 per-test init/shutdown 不可靠，不是 sleep 时间不够的问题。ROS2 官方也推荐 per-suite init。

2. **SHM transport 是第一大 Docker 陷阱**：`network_mode: host` 不够，必须加 `ipc: host`。否则所有节点在 ros2 graph 里可见但数据不通——非常隐蔽的故障模式。

3. **多阶段 Docker 构建的价值不仅在体积**：builder 阶段跑测试 → 阻塞构建失败，等于 CI gate 前置到了 `docker build` 命令。

4. **GWT 命名应该放在测试名里**：`TEST_F(Suite, NodeName_Given_Then)` — 失败时 ctest 直接告诉你哪个 Given-Then 条件破了，不需要翻代码。

5. **Podman 兼容成本**：短镜像名、socket 路径、docker-compose v1/v2 差异、部分 flag 不支持。如果团队统一用 Docker 就简单很多，但如果目标环境是 RHEL/Fedora，podman 兼容值得做。
